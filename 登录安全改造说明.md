# 登录安全改造说明

## 改造目标

本次改造解决的是一个具体问题：

- 原实现把 `access token` 明文放在 `localStorage`
- 同源下的其他 tab 会直接共享这份 token
- 结果是用户在 A tab 登录后，B tab 即使不是这次登录的发起方，也能直接带着 `access token` 调接口

这不是“双 token 机制失效”，而是会话粒度过粗。系统之前管理的是“用户 token”，现在改成“用户会话”。

## 核心设计

### 1. access token 不再持久化

- 前端不再把 `access token` 存进 `localStorage`
- `access token` 只存在当前页面内存中
- 页面刷新后，通过 `refresh token` 尝试恢复当前 tab 的会话

这样做以后，新的 tab 打开时不会天然继承旧 tab 的 `access token`。

### 2. refresh token 继续放在 HttpOnly Cookie

- `refresh token` 仍由后端通过 `HttpOnly Cookie` 下发
- 前端 JavaScript 不能直接读取它
- 这样可以避免把长寿命令牌暴露给脚本

### 3. 引入 tab 级别的 client instance id

- 前端为每个 tab 在 `sessionStorage` 中生成一个 `clientInstanceId`
- 该值通过请求头 `X-Client-Instance-Id` 传给后端
- 登录、刷新、登出都会校验这个值

由于 `sessionStorage` 是 tab 级别，新开的普通 tab 无法拿到旧 tab 的 `clientInstanceId`。

### 4. 后端从“用户级 token”改为“会话级 session”

现在后端新增三类数据结构：

- `user_security_state`
- `auth_sessions`
- `auth_refresh_tokens`

它们的职责分别是：

- `user_security_state` 负责账户级安全版本号，例如管理员踢人、未来密码修改后的全局失效
- `auth_sessions` 负责表示一次真实登录会话，并绑定 `clientInstanceId`
- `auth_refresh_tokens` 负责刷新令牌轮换、失效和复用检测

## 数据结构说明

### user_security_state

字段含义：

- `user_id`：用户 id
- `auth_version`：账户认证版本号，管理员强制下线时递增
- `credential_version`：预留给密码修改等场景
- `force_logout_after`：预留给按时间全局失效

### auth_sessions

字段含义：

- `id`：会话 id，JWT 的 `sid` 就是它
- `user_id`：所属用户
- `session_scope`：当前实现固定为 `tab`
- `client_instance_id`：当前 tab 的唯一标识
- `status`：会话状态
- `last_seen_at` / `idle_expires_at`：会话活跃与空闲时间
- `revoked_at` / `revoke_reason`：吊销信息
- `last_ip` / `user_agent`：审计信息

### auth_refresh_tokens

字段含义：

- `token_hash`：只存 refresh token 哈希，不存明文
- `session_id`：归属哪个会话
- `parent_id`：刷新轮换链路
- `used_at`：是否已经被消费
- `revoked_at`：是否被吊销
- `reuse_detected`：是否检测到旧 token 被重放

## 登录链路

1. 前端生成或读取当前 tab 的 `clientInstanceId`
2. 登录请求带上 `X-Client-Instance-Id`
3. 后端创建 `auth_sessions` 记录
4. 后端签发带 `sid` 和 `ver` 的 access token
5. 后端创建一条 `auth_refresh_tokens` 记录，并把 refresh token 明文写入 HttpOnly Cookie

## 刷新链路

1. 前端调用 `/api/v1/auth/refresh`
2. 浏览器自动带上 HttpOnly Cookie
3. 前端同时带上当前 tab 的 `X-Client-Instance-Id`
4. 后端校验：
   - refresh token 是否存在
   - refresh token 是否已失效或已被消费
   - session 是否还有效
   - `clientInstanceId` 是否与 session 绑定值一致
5. 校验通过后：
   - 当前 refresh token 标记为已消费
   - 新建下一条 refresh token
   - 重新签发 access token

如果一个旧 refresh token 被再次使用，会被识别为重放，后端会吊销整个会话。

## 鉴权链路

每次访问受保护接口时，后端不再只信任 JWT 本身，而是会继续校验：

- JWT 中的 `ver` 是否仍等于 `user_security_state.auth_version`
- JWT 中的 `sid` 是否对应一条仍然有效的 `auth_sessions`

这意味着：

- 管理员踢人后，旧 access token 会因为 `auth_version` 变化而失效
- 某个 session 被吊销后，带着旧 `sid` 的 access token 也会失效

## 针对“别的 tab 乱用 access token”的效果

改造后的行为是：

- A tab 登录后，`access token` 只存在 A tab 内存里
- B tab 不会再从 `localStorage` 拿到这份 token
- B tab 即使共享浏览器 cookie，也因为没有 A tab 的 `clientInstanceId`，无法刷新出新的 access token

所以普通新开 tab 不会再自动继承当前登录态。

## 仍然需要知道的边界

### 1. 这不是 XSS 的最终解

如果页面存在同源 XSS，恶意脚本仍可能直接在已登录 tab 内发起请求。所以还需要继续补：

- CSP
- 富文本和模板输出转义
- 依赖供应链审计

### 2. 浏览器“复制标签页”可能复制 sessionStorage

某些浏览器在“复制当前标签页”时会复制 `sessionStorage`。这会让复制出来的 tab 继承同一个 `clientInstanceId`。这属于浏览器行为边界，不是当前项目单独能完全消掉的。

### 3. 当前模型更偏向“当前活跃 tab”恢复

由于 refresh token 使用浏览器 cookie 承载，而 cookie 是浏览器共享的，所以如果用户在另一个 tab 主动重新登录，会覆盖旧 cookie。旧 tab 的 access token 会继续活到过期，但后续刷新会受影响。

如果后续你要支持“多个 tab 各自独立长期在线”，可以进一步升级成“浏览器会话 + tab proof”双层模型。

## 部署前需要做的事

1. 在 PostgreSQL 执行 `auth_security_upgrade.sql`
2. 生产环境把 `JWT_SECRET` 改为环境变量，不要继续使用默认值
3. 生产环境把 `jwt.refresh-cookie-secure` 改成 `true`
4. 回归测试登录、刷新、登出、管理员踢人、标签页切换
